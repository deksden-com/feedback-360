# Cross-epic scenarios (XE) — foundation
Status: Draft (2026-03-07)

Цель XE: дать системе **исполняемые сквозные сценарии**, которые проходят через несколько эпиков и проверяют продукт как единое целое, а не как набор отдельных фич.

Зафиксированные инварианты:
- `xe_run` — orchestration-level сущность прогона. Это “мир” сценария на уровне запуска, но **не** дополнительная колонка `world_id` во всех продуктовых таблицах.
- XE разрешены только в `local` и `beta`; в `prod` destructive XE-операции запрещены жёстко.
- Сценарии описываются **отдельно** от раннера: сценарные материалы лежат в каталоге `scenarios/`, а исполняющий код раннера — в отдельном приложении/пакете монорепо.
- Runtime state раннера хранится в файловом workspace прогона; БД хранит только registry/summary/run lifecycle.
- Есть `system seed` и далее именованные `seed`-наборы. Отдельную сущность “XE seed” не вводим.
- `seed` для сценария может строиться поверх `system seed`.
- По умолчанию конкурентные XE-runs не поддерживаем; на `beta` допускается только один активный run.
- Упавшие runs сохраняются для расследования; cleanup запускается явной CLI-командой или по policy для expired runs.
- Фазы исполняются детерминированным кодом раннера, а не Vitest как первичным движком.
- Проверки фазы и итоговые проверки сценария — это один и тот же механизм assertions; отдельную runtime-сущность “final assertions” не вводим.

Роли XE:
- сценарий = спецификация (fixtures, phases, expected outcomes, artifacts);
- раннер = исполняет фазы, пишет state/artifacts, собирает assertions;
- AI-агент = верхнеуровневый оркестратор, который использует CLI/раннер и оформляет evidence.
