# MBB — Duo pattern (summary + details)
Status: Updated (2026-03-09)

`Duo pattern` — это основной способ декомпозиции сложной документации в `feedback-360`: сначала короткий обзорный документ, затем детальные файлы по отдельным аспектам. Это помогает держать SSoT компактным, не смешивать аудитории и не плодить дубли.

## Когда применять

Используем duo pattern, когда:
- документ начинает смешивать несколько самостоятельных тем;
- появляется разная глубина чтения (`quick understanding` vs `deep implementation context`);
- один и тот же файл нужен сразу разным аудиториям (onboarding, implementation, review, operations);
- документ становится трудно обновлять без случайного дублирования.

## Tiers (правило размера)

### Tier 1 — до ~250 строк
- Файл обычно остаётся цельным.
- Подходит для узкой темы, простого решения, точечного how-to.

### Tier 2 — ~250–800 строк
- Предпочтителен разбор на summary + 1–2 detail файла.
- Обычно хватает структуры вида:
  - `<topic>.md`
  - `<topic>-implementation.md`
  - `<topic>-examples.md` или `<topic>-edge-cases.md`

### Tier 3 — 800+ строк
- Декомпозиция обязательна.
- Часто появляется `index.md` + обзор + несколько detail-файлов.
- Если тема продолжает расти, detail-файл сам может стать новым summary-слоем для под-темы.

Это правило не про "магическое число строк", а про читаемость и поддерживаемость. Если файл на 180 строк уже трудно держать в голове — дробим раньше.

## Роли файлов

### Summary file
Обычно называется `<topic>.md`.

Его задача:
- объяснить, что это за тема;
- зафиксировать ключевые инварианты;
- дать минимальный quick-start context;
- направить на detail-файлы аннотированными ссылками.

Summary не должен пытаться повторить весь detail-content. Его ценность — дать карту темы.

### Detail files
Обычно называются `<topic>-<aspect>.md`.

Их задача:
- раскрыть отдельный аспект темы;
- держать детали в одном месте;
- не дублировать соседние detail-файлы;
- ссылаться обратно на summary.

## Типичные суффиксы

- `-architecture` — архитектурная картина, границы, взаимодействия
- `-implementation` — детали реализации, seam points, invariants в коде
- `-api` — контракты и операции
- `-examples` — практические сценарии и use cases
- `-edge-cases` — tricky behavior, ограничения, corner cases
- `-testing` — test strategy, fixtures, acceptance logic
- `-troubleshooting` — диагностика и восстановление
- `-operations` — release/runbook/ops-specific нюансы

Не нужно использовать суффиксы формально. Они нужны там, где реально помогают быстро понять роль файла.

## Базовые структуры

### Вариант 1 — summary + детали

```text
spec/domain/results/
  index.md
  results.md
  results-calculations.md
  results-visibility.md
  results-edge-cases.md
```

### Вариант 2 — overview в index

```text
spec/ui/screens/
  index.md
  internal-home.md
  hr-employees.md
  questionnaire-fill.md
```

Здесь `index.md` уже играет роль общего summary по группе, а отдельные screen docs — атомарные документы.

### Вариант 3 — detail-файл сам становится summary

```text
spec/testing/
  xe-foundation.md
  xe-run-model.md
  xe-json-schemas.md
  xe-cli-contract.md
```

Здесь `xe-foundation.md` — верхний обзор, а `xe-run-model.md` может сам разрастись в маленькую тему с собственным duo-слоем.

## Правила summary-файла

Summary-файл должен:
- объяснять тему сверху вниз;
- содержать только устойчивые тезисы и ключевые решения;
- включать аннотированные ссылки на детали;
- не копировать примеры/таблицы/списки из detail-файлов без необходимости.

Хороший паттерн:
- что это;
- зачем нужно;
- ключевые инварианты;
- структура темы;
- ссылки на детали;
- related docs.

## Правила detail-файлов

Detail-файл должен:
- иметь чёткий аспект и границу;
- ссылаться на summary или index;
- не дублировать соседние detail-файлы;
- быть полезным как самостоятельный deep dive.

Если detail-файл становится фактическим дублём summary — это сигнал на пересборку структуры.

## Duo pattern и индексы

Если у темы появляется 3+ файлов, почти всегда нужен `index.md` папки.

`index.md` отвечает за:
- карту файлов;
- объяснение, как читать тему;
- порядок углубления (`start here -> then go deeper`).

Duo pattern без индекса допустим только для очень маленьких наборов файлов.

## Duo pattern и C4

В `feedback-360` duo pattern используется поверх C4-структуры:
- `L1` документы обычно остаются summary-first;
- `L2` контейнеры часто имеют `index.md` + contract/architecture docs;
- `L3` компоненты и feature areas чаще всего требуют duo-подхода, если тема выходит за рамки одного атомарного файла.

## Практические признаки, что пора дробить

Нужно декомпозировать, если:
- в документе появилось несколько секций уровня "это уже отдельная тема";
- файл регулярно читают по разным кускам, а не целиком;
- изменения в одной части часто случайно ломают другую;
- в документе трудно понять, что нормативно, а что просто пояснение;
- индекс папки уже не может коротко объяснить, зачем читать этот файл.

## Антипаттерны

Плохо:
- огромный файл с архитектурой, implementation details, тестами и runbook одновременно;
- detail-файлы, которые повторяют по 70% summary;
- naming без системы (`foo-notes.md`, `foo-more.md`, `foo-final.md`);
- создание duo-структуры ради красоты, когда тема всё ещё атомарная.

## Мини-чеклист при декомпозиции

- [ ] Определён один summary-entrypoint
- [ ] У каждой detail-части есть свой аспект
- [ ] Обновлён `index.md`
- [ ] Добавлены аннотированные ссылки между уровнями
- [ ] Убраны дубли, а не просто размножены файлы
- [ ] Если менялись роли документов — синхронизированы frontmatter / related links / templates
