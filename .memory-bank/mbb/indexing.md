# MBB — Indexing
Status: Draft (2026-03-03)

Индексный файл — это `index.md` в папке, который:
- объясняет назначение папки,
- перечисляет документы с аннотированными ссылками,
- помогает ИИ-агентам и людям находить SSoT.

## Types
- **Shallow index**: описывает только файлы текущей папки. Используем, когда структура плоская и простая.
- **Deep index**: даёт навигацию на 1–2 уровня вниз. Используем, когда в папке много подпапок/тем.
- **Hybrid**: shallow + deep секции, если часть папки плоская, а часть — иерархичная.

## Meta-groups in parentheses
Папки в скобках `(like-this)/` считаются логическими группировками, а не отдельными контейнерами. В индексе нужно явно сказать, что это grouping, и куда ведёт (к каким “настоящим” контейнерам/темам).

## Annotation format
Формат ссылки в индексе:
- `[Display name](relative/path.md): <что внутри>. <зачем читать>.`

## Root index strategy (`.memory-bank/index.md`)
Главный индекс — это “entrypoint” для агента. Его задача не перечислить всё, а быстро вывести к критичным SSoT документам.

Что обязательно должно быть:
- `Quick start` (5–10 ссылок): структура проекта, системный overview, glossary, coding/architecture rules, delivery+git/deploy правила.
- `Key folders`: ссылки на индексы верхнего уровня (`spec/`, `plans/`, `adr/`, `mbb/`) как карту SSoT.
- `Important docs by area`: частичное раскрытие вложенных индексов на **2–3 уровня** (например `spec/domain/*`, `plans/*`) только по самым важным документам.

## 2–3 levels disclosure rule
Для главного индекса используем curated deep index:
- Разворачиваем не более 2–3 уровней навигации.
- Показываем документы, которые нужны агенту “в первые минуты” работы и при закрытии фич.
- Не дублируем полный список файлов из подпапок; для полного перечня всегда даём ссылку на индекс папки.

## Selection heuristics (what is “important”)
В главный индекс попадают документы, которые:
1) содержат инварианты/ограничения (можно сломать систему, если пропустить),
2) задают процесс delivery (commit/PR/acceptance/evidence),
3) нужны для регулярных задач (реализация фич, тесты, деплой),
4) ускоряют онбординг нового агента.

Если документ редко нужен в ежедневной работе — оставляем его в индексах подпапок, но не перегружаем им главный индекс.

## UI/XE indexing rule
Для контуров, которые быстро растут и связаны с automation, используем обязательные подиндексы:
- `spec/ui/screens/index.md`
- `spec/ui/pom/index.md`
- `plans/xe/index.md`

Это нужно, чтобы агент мог быстро найти:
- нормативный screen contract,
- automation/POM contract,
- конкретный XE scenario.

## Status/evidence sync hook
- Когда индекс или epic progress report меняется после закрытия FT, рядом должно быть ясно, каким audit это подтверждается (`pnpm docs:audit` или эквивалент).
- Идея простая: индексы — не “ручной отчёт”, а проверяемая навигация по актуальному состоянию.
