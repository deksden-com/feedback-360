# MBB — Indexing
Status: Updated (2026-03-09)

`index.md` — это entrypoint папки. Хороший индекс не перечисляет файлы ради перечисления, а помогает быстро понять: что это за раздел, куда идти сначала и где находится нормативный SSoT.

## Цель индексов

Индекс должен:
- объяснять назначение папки;
- показывать структуру на правильной глубине;
- давать аннотированные ссылки;
- предотвращать orphan files;
- ускорять работу людей и агентов.

## Типы индексов

### Shallow
Описывает в основном файлы текущей папки.

Используем, когда:
- папка простая;
- файлов немного;
- глубокой иерархии почти нет.

### Deep
Показывает 1–2 уровня вниз.

Используем, когда:
- папка большая;
- есть подпапки с собственными темами;
- нужно быстро провести читателя по ветке документации.

### Hybrid
Часть разделов shallow, часть deep.

Это самый частый вариант для верхних индексов (`.memory-bank/index.md`, `spec/index.md`, `plans/index.md`).

## Формат аннотированной ссылки

Каждая ссылка в индексе оформляется так:

```md
- [Display name](path/to/file.md): что внутри. зачем читать.
```

Идеальный минимум:
- 1 предложение о содержимом;
- 1 предложение о пользе.

## Что должно быть в хорошем индексе

### 1. Короткое назначение папки
В 1–3 предложениях:
- что это за раздел;
- какой у него тип знания (`spec`, `guides`, `plans`, `adr`, `assets`, `mbb`);
- зачем сюда заходить.

### 2. Навигация по ключевым файлам
Не всё подряд, а те файлы, которые реально нужны как entrypoints.

### 3. Правильная глубина
- не прячем важный SSoT слишком глубоко;
- но и не делаем из индекса dump всех файлов.

## Правило 2–3 уровней для корневого индекса

`.memory-bank/index.md` должен:
- дать quick start;
- показать ключевые разделы;
- частично раскрыть важные документы на 2–3 уровня вниз;
- не дублировать полные каталоги подпапок.

Главный индекс — это curated map, а не файловый листинг.

## Heuristics: что поднимать в индекс выше

В верхние индексы поднимаем документы, которые:
- содержат инварианты и ограничения;
- нужны для ежедневной реализации фич;
- нужны перед merge/release;
- ускоряют onboarding нового агента;
- часто используются как source of truth для других документов.

## Meta-groups и скобки

Папки в скобках вроде `(group)/` — это логические grouping layers. Индекс должен явно говорить, что это не отдельный runtime/container level, а просто удобная группировка документов или пакетов.

## Индексы для UI и XE

Для UI/XE-подсистем у нас обязательны отдельные навигационные entrypoints:
- `spec/ui/index.md`
- `spec/ui/screens/index.md`
- `spec/ui/pom/index.md`
- `plans/xe/index.md`
- `spec/testing/index.md`

Это нужно, чтобы быстро находить:
- screen contract;
- `screen_id` / `testIdScope`;
- POM mapping;
- XE scenario docs;
- scenario/runtime specs.

## Индекс и status/evidence sync

Если после реализации фичи меняется:
- статус epic/feature;
- verification/evidence link;
- навигация по новым документам;

индекс должен обновиться в том же changeset. Иначе появляется orphan-или-устаревшая навигация.

## Практический чеклист индекса

- [ ] Папка объяснена как раздел знаний
- [ ] Есть аннотированные ссылки, а не просто list of files
- [ ] Глубина раскрытия соответствует сложности раздела
- [ ] Новые документы не остались orphan
- [ ] Если есть screen/spec/evidence документы — они discoverable из нужного индекса
- [ ] Индекс не дублирует содержимое соседних индексов без причины

## Антипаттерны

Плохо:
- индекс = просто список файлов без пояснений;
- индекс повторяет содержание каждого документа почти verbatim;
- важный SSoT buried в под-под-папке без ссылки из верхнего уровня;
- новый документ добавили в папку, но забыли про индекс;
- индекс устарел и больше не отражает реальную карту раздела.
