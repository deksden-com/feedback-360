# MBB — C4 in docs
Status: Updated (2026-03-09)

Мы используем C4 не как формальный ритуал, а как способ держать документацию и архитектуру в одном языке: от системы целиком до отдельных компонентов и экранов.

## Как C4 применяется в `feedback-360`

### L1 — System level
Это документы, которые объясняют систему целиком:
- что за продукт;
- кто его пользователи;
- какие внешние системы участвуют;
- какие крупные ограничения и границы есть у решения.

Примеры у нас:
- `spec/project/system-overview.md`
- `spec/c4/l1-system-context.md`
- `spec/glossary.md`

L1 нужен для быстрого доменного и архитектурного grounding.

### L2 — Container / subsystem level
Это документы про крупные части системы и их границы:
- `web app`
- `core`
- `db`
- `auth`
- `notifications`
- `AI`
- `XE runner`
- группы документов вроде `spec/project/`, `spec/security/`, `spec/operations/`, `spec/ui/`

Примеры:
- `spec/c4/l2-containers.md`
- `spec/project/index.md`
- `spec/ui/index.md`
- `spec/testing/index.md`
- `spec/operations/index.md`

L2 отвечает на вопрос: какие крупные контейнеры/подсистемы есть и как они взаимодействуют.

### L3 — Component / feature area level
Это уровень конкретных модулей, экранов, процессов и feature areas.

Примеры:
- `spec/project/feature-area-boundaries.md`
- `spec/ui/screens/*`
- `spec/ui/pom/*`
- `spec/testing/xe-run-model.md`
- `spec/domain/results/*`

L3 отвечает на вопрос: как устроен конкретный кусок системы, какие у него контракты, инварианты и точки интеграции.

### L4 — Code level
Это уже код. Мы не делаем отдельную C4-документацию на каждую функцию, но поддерживаем двустороннюю трассировку:
- docs → code через file references;
- code → docs через JSDoc теги и screen/test-id contracts.

## Как раскладывать документы по C4

### Для `spec/`
- `spec/project/*`, `spec/c4/*` — в основном L1/L2;
- `spec/domain/*`, `spec/security/*`, `spec/testing/*`, `spec/ui/*` — обычно L2/L3;
- `spec/ui/screens/*` — почти всегда L3;
- `spec/ui/design-system/*` — L2/L3 foundations.

### Для `plans/`
`plans/` не заменяет C4, но должен на него опираться.

- epic обычно привязан к нескольким L2/L3 зонам;
- feature должна явно ссылаться на те L2/L3 документы, которые затрагивает;
- XE/GS сценарии часто проходят поперёк нескольких контейнеров, но всё равно должны ссылаться на соответствующие screen specs / operation catalogs / domain rules.

### Для `guides/`
`guides/` — не архитектурный SSoT, но их полезно маркировать screen IDs и опираться на screen registry. По сути:
- tutorial/how-to чаще всего описывает пользовательский проход через несколько L3 screen contracts;
- explanation может объяснять, как несколько L2/L3 частей складываются в единый пользовательский опыт.

## Meta-groups в скобках

Папки вида `(something)/` используются как логические группировки, а не как отдельный C4 контейнер.

Пример мышления:
- `guides/tutorials/` — это тип документации, а не новый контейнер продукта;
- `assets/ui/` — это хранилище референсов, а не часть runtime architecture.

Индекс таких групп должен прямо говорить, что это grouping layer.

## Что документ какого уровня должен отвечать

### L1 документ
Должен отвечать:
- что это за система;
- кто в ней участвует;
- какие крупные интеграции и ограничения есть.

### L2 документ
Должен отвечать:
- за что отвечает подсистема;
- что у неё за границы;
- с чем она взаимодействует;
- какие у неё ключевые контракты.

### L3 документ
Должен отвечать:
- как устроен конкретный компонент/экран/feature area;
- какие у него инварианты;
- какие действия и ограничения у него есть;
- где его точки интеграции.

## Практические правила для нас

- Если документ нельзя быстро отнести хотя бы к одному уровню — он, скорее всего, смешивает несколько тем.
- Если документ пытается быть одновременно L1 и L3 — применяем duo pattern.
- Если новый epic меняет границы контейнера или feature area, это должно отразиться в L2/L3 spec, а не только в планах.
- Если меняется экран, обновляются `screen-registry`, `screen spec`, `test-id registry`, guides/evidence — это наш UI-вариант C4-трассировки.

## Мини-чеклист

При создании нового документа:
- [ ] Ясно, это L1 / L2 / L3?
- [ ] Понятно, в какой индекс он должен попасть?
- [ ] Понятно, это нормативный spec, guide, ADR или plan?
- [ ] Есть ли рядом документ другого уровня, на который нужно сослаться вместо дублирования?
